## Проект: Консольное приложение «CLI-Метеоролог» (Weather CLI App)

**Описание:** Разработка консольной утилиты для получения текущей погоды и прогноза на несколько дней для любого города мира. Проект учит делать HTTP-запросы, парсить сложные JSON-ответы, обрабатывать ошибки сети, работать с аргументами командной строки и сохранять историю действий пользователя в файл.

### 🌐 Используемый API

Для этого проекта будет использоваться **Open-Meteo API**. Это отличный инструмент для обучения, так как он абсолютно бесплатен, работает без регистрации и **не требует API-ключей или токенов**.

* **Geocoding API** (поиск координат по названию города): [https://open-meteo.com/en/docs/geocoding-api](https://open-meteo.com/en/docs/geocoding-api)
* **Weather API** (получение погоды по координатам): [https://open-meteo.com/en/docs](https://open-meteo.com/en/docs)

### 🛠 Стек технологий и библиотеки

**Встроенные библиотеки Python:**

* `json` — для работы с историей запросов (сохранение/чтение).
* `argparse` — для создания удобного интерфейса командной строки (чтение флагов и аргументов).
* `datetime` — для форматирования времени и дат из ответов API.
* `pathlib` — для безопасной работы с путями к файлам (сохранение логов/истории).

**Сторонние библиотеки (требуют `pip install`):**

* `requests` — основной инструмент для выполнения GET-запросов к API.
* `rich` *(опционально)* — для красивого форматирования вывода в консоль (цвета, таблицы, иконки).

### 🎯 Основные цели и задачи разработчика

Проект должен быть разбит на логические модули, а не написан в одном файле `main.py`. Это ключевой навык для Junior-специалиста.

**1. Работа с сетью (API Client):**

* Реализовать функцию поиска координат (широта и долгота) по названию города через Geocoding API.
* Реализовать функцию получения погоды по найденным координатам.
* **Важно:** Корректно обработать ситуации, когда город не найден или отсутствует интернет-соединение (использовать блоки `try...except` и исключения библиотеки `requests`).

**2. Интерфейс командной строки (CLI):**

* Настроить `argparse` так, чтобы утилиту можно было вызывать с параметрами.
* *Пример:* `python weather.py --city "Москва" --days 3`


* Добавить флаг `--history` для вывода последних 5 запрошенных городов.

**3. Работа с данными и файлами:**

* После каждого успешного запроса сохранять данные (название города, дата запроса, температура) в локальный файл `history.json`.
* При запуске с флагом `--history` читать этот файл и выводить данные пользователю.

**4. Парсинг и форматирование:**

* Извлечь из "сырого" JSON-ответа API только нужные данные (текущая температура, скорость ветра, осадки).
* Преобразовать форматы дат API (ISO 8601) в читаемый для пользователя вид с помощью `datetime`.

### 📂 Рекомендуемая структура проекта

```text
weather_project/
│
├── api_client.py       # Логика запросов к Open-Meteo (get_coordinates, get_weather)
├── storage.py          # Логика сохранения и чтения history.json
├── formatter.py        # Функции для красивого вывода данных в консоль
├── main.py             # Точка входа, настройка argparse и связывание модулей
├── requirements.txt    # Список зависимостей (requests, rich)
└── .gitignore          # Исключение кэша и файлов IDE

```

### 🚀 Дополнительные задания "со звездочкой" (Level Up)

*Если студент справился с основной частью, предложите ему улучшить код:*

1. **Типизация:** Добавить Type Hints (аннотации типов) для всех функций и методов.
2. **ООП:** Обернуть логику работы с API и историей в классы (`WeatherAPI`, `HistoryManager`).
3. **Кэширование:** Если пользователь запрашивает погоду в одном и том же городе с разницей менее 15 минут, не делать запрос к API, а отдавать сохраненный локальный результат.
---------------
Для middle-разработчика требования должны быть заметно выше. Если junior учится писать работающий код и взаимодействовать с сетью, то middle должен показать умение работать с асинхронностью, проектировать базу данных, выстраивать архитектуру приложения и управлять потоками данных.

Поскольку мы исключаем веб-фреймворки (FastAPI/Django), идеальным форматом будет **асинхронный фоновый сервис (демон)** с интерфейсом командной строки для управления и аналитики.

## Проект: Асинхронный агрегатор рыночных данных (Async Market Data Aggregator)

**Описание:** Разработка фонового приложения, которое с заданной периодичностью параллельно опрашивает несколько публичных финансовых API, агрегирует данные о курсах валют/криптовалют и сохраняет их в реляционную базу данных. Приложение также включает CLI-инструмент для генерации аналитических отчетов на основе собранных данных.

### 🌐 Используемые API (Публичные, без токенов)

Проект будет собирать данные из двух независимых источников для демонстрации конкурентных запросов:

1. **CoinGecko API** (данные о рынке): [https://api.coingecko.com/api/v3/coins/markets?vs_currency=usd](https://api.coingecko.com/api/v3/coins/markets?vs_currency=usd)
2. **Binance Public API** (тикеры 24 часа): [https://api.binance.com/api/v3/ticker/24hr](https://api.binance.com/api/v3/ticker/24hr)

### 🛠 Стек технологий и библиотеки

**Встроенные библиотеки Python:**

* `asyncio` — ядро проекта для асинхронной работы.
* `logging` — для правильного логирования работы фонового процесса.
* `argparse` — для создания CLI-интерфейса аналитики.

**Сторонние библиотеки (требуют `pip install`):**

* `aiohttp` — для выполнения асинхронных HTTP-запросов.
* `SQLAlchemy` — ORM для проектирования схем и взаимодействия с базой данных.
* `alembic` *(опционально)* — для управления миграциями базы данных.
* *База данных:* `SQLite` (достаточно для тестового задания, но архитектура должна позволять легко переключиться на PostgreSQL).

### 🎯 Основные цели и задачи разработчика

**1. Асинхронность и конкурентность:**

* Реализовать параллельный опрос обоих API с использованием `asyncio.gather` или `asyncio.TaskGroup` (для Python 3.11+).
* Настроить таймауты и повторные попытки (retry pattern) на случай временной недоступности API.

**2. Проектирование базы данных (ORM):**

* Спроектировать схему БД с таблицами: `Asset` (информация об активе) и `PriceHistory` (история цен с привязкой ко времени и источнику).
* Настроить связи между таблицами (One-to-Many).
* Использовать асинхронный движок SQLAlchemy (`async_engine`).

**3. Архитектура фонового процесса (Сборщик):**

* Создать бесконечный цикл (event loop), который запускает процесс сбора данных каждые *N* минут.
* Корректно обрабатывать сигналы завершения работы (Graceful shutdown), чтобы не повредить данные при остановке скрипта.

**4. Аналитический CLI-инструмент:**

* Создать отдельную команду для запроса аналитики из БД.
* *Пример:* `python cli.py report --asset BTC --period 24h` должно выводить среднюю, максимальную и минимальную цену актива за указанный период, сравнивая данные с Binance и CoinGecko.

### 📂 Рекомендуемая структура проекта

```text
market_aggregator/
│
├── core/
│   ├── config.py         # Настройки приложения (периодичность, URL БД)
│   └── logger.py         # Настройки логирования
├── database/
│   ├── models.py         # Модели SQLAlchemy
│   └── session.py        # Инициализация асинхронной сессии БД
├── fetchers/
│   ├── base.py           # Базовый абстрактный класс для API-клиентов
│   ├── binance.py        # Клиент Binance API
│   └── coingecko.py      # Клиент CoinGecko API
├── worker.py             # Главный цикл фонового сбора данных
├── cli.py                # Интерфейс для генерации отчетов из БД
└── requirements.txt      # Зависимости

```

### 🚀 Дополнительные задания "со звездочкой" (Level Up)

* **Тестирование:** Покрыть бизнес-логику и API-клиенты тестами с использованием `pytest` и `pytest-asyncio` (использовать моки для сети).
* **Контейнеризация:** Написать `Dockerfile` для запуска воркера и `docker-compose.yml`, включающий контейнер с PostgreSQL.
* **Инверсия зависимостей (DI):** Реализовать слабую связность компонентов (например, передавать сессию БД и клиенты API через конструкторы).

## Проект: Событийно-ориентированный фреймворк для бэктестинга (Event-Driven Backtesting Engine)

**Описание:** Разработка легковесного движка для тестирования торговых алгоритмов на исторических данных. Проект требует реализации сложной архитектуры (паттерн Observer), строгой типизации, управления памятью (обработка огромных массивов данных без утечек) и распараллеливания вычислений для оптимизации параметров.

### 🌐 Используемый API

Для получения исторических данных (свечей/баров) будет использоваться публичный эндпоинт Binance, который отдает историю торгов без авторизации.

* **Binance Klines API** (исторические данные): [https://api.binance.com/api/v3/klines](https://www.google.com/search?q=https://api.binance.com/api/v3/klines)

### 🛠 Стек технологий и инструменты

**Встроенные библиотеки Python:**

* `typing` — продвинутое использование (Generics, Protocols) для создания строгих интерфейсов.
* `multiprocessing` — для параллельного прогона множества стратегий (Grid Search оптимизация).
* `collections` и `itertools` — для эффективной работы с очередями событий и комбинаторики.
* `weakref` — для управления ссылками и предотвращения утечек памяти в кэшах.
* `cProfile` и `tracemalloc` — встроенные инструменты для профилирования производительности и памяти.

**Сторонние библиотеки:**

* Проект должен быть реализован **максимально на чистом Python** без использования тяжелых фреймворков вроде Pandas для обработки данных. Это заставит кандидата думать об алгоритмической сложности и структурах данных.

### 🎯 Основные цели и задачи разработчика

**1. Проектирование архитектуры (Design Patterns):**

* Разработать событийную шину (Event Bus). Движок должен работать на основе событий: `MarketEvent` (новые данные), `SignalEvent` (решение стратегии), `OrderEvent` (отправка заявки), `FillEvent` (исполнение заявки).
* Реализовать паттерн "Стратегия" (Strategy), предоставив базовый абстрактный класс `BaseStrategy`, от которого пользователи фреймворка будут наследовать свои алгоритмы.

**2. Оптимизация памяти (Memory Management):**

* Реализовать "ленивую загрузку" (Lazy Loading) исторических данных с помощью генераторов. Движок должен уметь обрабатывать гигабайты JSON-данных, сохраняя потребление оперативной памяти в пределах строгих лимитов (например, не более 100 МБ).
* Использовать `__slots__` в дата-классах (событиях и тиках) для снижения накладных расходов на создание миллионов объектов.

**3. Конкурентность и распараллеливание:**

* Создать модуль `Optimizer`, который принимает стратегию и диапазоны ее параметров (например, период скользящей средней от 10 до 50).
* Использовать `multiprocessing.Pool` или `ProcessPoolExecutor` для распределения независимых прогонов бэктеста по ядрам процессора для ускорения вычислений.

**4. Разработка API (Developer Experience):**

* Код должен быть спроектирован как библиотека (package). Главная задача — сделать интерфейс написания кастомных стратегий интуитивно понятным и лаконичным для конечного пользователя.

### 📂 Рекомендуемая структура проекта

```text
backtrader_engine/
│
├── engine/
│   ├── event_bus.py      # Обработчик очереди событий
│   ├── data_feed.py      # Генераторы для скачивания и стриминга данных по API
│   ├── execution.py      # Симулятор биржевого стакана и исполнения заявок
│   └── portfolio.py      # Учет баланса, комиссий и расчет PnL
├── strategy/
│   ├── base.py           # Абстрактные классы (BaseStrategy)
│   └── examples.py       # Примеры реализованных торговых алгоритмов
├── optimization/
│   └── grid_search.py    # Логика мультипроцессорной оптимизации параметров
├── main.py               # Точка входа для запуска тестов
└── pyproject.toml        # Конфигурация пакета и требования

```

### 🚀 Дополнительные задания "со звездочкой" (Level Up)

* **Профилирование:** Предоставить текстовый отчет с результатами работы `cProfile`, показывающий "узкие места" алгоритма до и после применения оптимизаций (например, замены стандартных списков на `collections.deque`).
* **Cython / C-Extensions:** Переписать самый нагруженный модуль (например, функцию расчета скользящей средней или симулятор сведения ордеров) на Cython для получения скорости, сравнимой с C++.
* **Упаковка:** Оформить проект как полноценный pip-пакет, готовый к публикации на PyPI (настройка `setup.py` / `pyproject.toml`).
