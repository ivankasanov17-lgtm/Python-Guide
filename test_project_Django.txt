Переход к Django — это важнейший этап для junior-разработчика. Если в чистом Python они учатся алгоритмам и структурам данных, то в Django они впервые сталкиваются с полноценной веб-архитектурой (MVT/MVC), базами данных и клиент-серверным взаимодействием.

Для уровня Junior критически важно не перегрузить их сложным фронтендом или микросервисами, а дать прочувствовать «магию» Django — его встроенные батарейки (админку, ORM, формы и систему авторизации).

Ниже представлено описание идеального первого веб-проекта, а также интерактивная схема, которая поможет студентам визуально понять, как устроен жизненный цикл запроса в Django.

## Проект: Веб-приложение «TaskFlow: Семейный таск-трекер» (Family Task Tracker)

**Описание:** Классический многопользовательский таск-менеджер. Проект является «золотым стандартом» для старта в веб-разработке. Студент научится проектировать связанные таблицы в БД через Django ORM, настраивать регистрацию и авторизацию пользователей, работать с формами и выводить динамические данные в HTML-шаблоны.

### 🌐 Используемый API

Для создания интерактивности на главной странице приложения будет выводиться случайный «совет дня» для продуктивности, получаемый из публичного API. Это научит студента интегрировать сторонние HTTP-запросы прямо на стороне бэкенда Django.

* **Advice Slip JSON API** (случайные советы): [https://api.adviceslip.com](https://api.adviceslip.com)

### 🛠 Стек технологий и библиотеки

* **Django** (версии 4.2+ или 5.0+).
* **SQLite** (по умолчанию встроен в Django, идеально для старта).
* **requests** — для синхронного запроса к API советов из Django View.
* **Bootstrap 5** (через CDN) — для быстрого создания аккуратного интерфейса без глубокого погружения в CSS.

### 🎯 Основные цели и задачи разработчика

**1. Проектирование моделей данных (Django ORM):**

* Создать модель `Task` со следующими полями: заголовок, описание, дата создания, дедлайн, статус (выполнено/нет), приоритет и связь с пользователем.
* Реализовать связь «Один-ко-многим» (Foreign Key) между встроенной моделью пользователя `User` и моделью `Task` (пользователь может иметь много задач, но задача принадлежит одному автору).

**2. Система авторизации (Django Auth):**

* Использовать встроенные механизмы Django для регистрации пользователей, входа и выхода из системы.
* **Важно:** Закрыть доступ к страницам создания и редактирования задач для неавторизованных гостей (использовать декоратор `@login_required` или миксин `LoginRequiredMixin`).

**3. CRUD-операции и Формы (Django Forms):**

* Реализовать просмотр списка задач, детальный просмотр, создание, редактирование и удаление (классический CRUD).
* Использовать `ModelForm` для автоматической генерации и валидации HTML-форм на основе полей модели `Task`.

**4. Шаблоны и логика (Django Templates):**

* Настроить базовый шаблон `base.html` с навигационной панелью (меню) и использовать наследование шаблонов.
* Реализовать фильтрацию задач на странице: «Все задачи», «Только мои», «Выполненные», «Просроченные».

### 🚀 Дополнительные задания "со звездочкой" (Level Up)

1. **Кастомизация админки:** Настроить `admin.py` так, чтобы в списке задач в панели администратора были видны колонки автора, дедлайна и статуса, а также работали фильтры по пользователям.
2. **Django Signals:** Написать сигнал, который при регистрации нового пользователя автоматически создает для него приветственную задачу-инструкцию.
3. **API интеграция через JS (AJAX):** Сделать так, чтобы статус задачи (выполнено/активно) менялся по клику на чекбокс без полной перезагрузки страницы (с использованием Fetch API на фронтенде и JSONResponse на бэкенде Django).

### 🎓 Интерактивный симулятор: Архитектура Django MVT (Model-View-Template)

Самая большая сложность для Junior-разработчика при переходе на Django — понять, как запрос от пользователя проходит через проект и возвращается обратно в виде веб-страницы.

Используйте этот симулятор, чтобы наглядно показать студенту, как взаимодействуют маршруты (`urls.py`), контроллеры (`views.py`), база данных (`models.py`) и шаблоны (`templates`).
=============
Для уровня Middle-разработчика классический MVT (шаблоны и базовые формы) уже пройден. Здесь фокус смещается на создание **REST API (DRF)**, работу с **асинхронными фоновыми задачами (Celery + Redis)**, оптимизацию запросов к базе данных и кэширование.

Проект должен заставить разработчика думать о производительности системы под нагрузкой и правильном разделении обязанностей между веб-сервером и фоновыми воркерами.

## Проект: Асинхронный агрегатор космических новостей «SpaceNews Hub»

**Описание:** REST API сервис, который в фоновом режиме собирает статьи, новости и отчеты о космических запусках из внешнего источника, структурирует их, кэширует частые запросы и предоставляет клиентам удобный API с гибкой фильтрацией, полнотекстовым поиском и ограничением частоты запросов (Rate Limiting).

### 🌐 Используемый API

Для сбора данных используется **Spaceflight News API** — стабильный, бесплатный и не требующий авторизации публичный ресурс:

* **Документация API:** [https://spaceflightnewsapi.net](https://spaceflightnewsapi.net)
* **Эндпоинт для сбора статей:** [https://api.spaceflightnewsapi.net/v4/articles/](https://www.google.com/search?q=https://api.spaceflightnewsapi.net/v4/articles/)

### 🛠 Стек технологий и библиотеки

* **Django 5.x** и **Django REST Framework (DRF)** — основа проекта.
* **Celery** — для управления очередью фоновых задач.
* **Redis** — в качестве брокера сообщений для Celery и кэш-бэкенда для Django.
* **django-filter** — для удобной фильтрации на эндпоинтах.
* **pytest-django** — для написания интеграционных тестов.
* **PostgreSQL** — рекомендуется в качестве основной СУБД для демонстрации работы с индексами и оптимизации.

### 🎯 Основные цели и задачи разработчика

**1. Проектирование REST API (DRF):**

* Создать сериализаторы (`serializers.py`) для валидации и трансформации данных новостей.
* Реализовать эндпоинты для получения списка новостей с поддержкой пагинации (разбивки на страницы).
* Настроить фильтрацию по источнику (news site), дате публикации и полнотекстовый поиск по заголовкам.
* Настроить ограничение количества запросов (`Throttle Classes`) для защиты API от перегрузки.

**2. Интеграция фоновых задач (Celery + Redis):**

* Написать периодическую задачу (`shared_task`), которая запускается каждые 10 минут (используя `django-celery-beat`) и запрашивает новые статьи из Spaceflight News API.
* Реализовать защиту от дублирования: в базу данных должны записываться только те статьи, которых там еще нет (проверка по уникальному ID внешней статьи).
* Настроить механизм повторных попыток выполнения задачи (`retry`) при ошибках сети.

**3. Оптимизация БД и Кэширование:**

* Избежать проблемы N+1 запросов при получении связанных данных (например, авторов или тегов статей) с помощью `select_related` и `prefetch_related`.
* Добавить индексы в базу данных на поля, по которым чаще всего происходит фильтрация и поиск.
* Настроить кэширование списка популярных новостей в Redis. При добавлении новых статей фоновым воркером кэш должен инвалидироваться (сбрасываться).

**4. Тестирование:**

* Написать тесты, проверяющие парсинг внешнего API (используя `requests-mock` или `responses` для изоляции от реальной сети).
* Покрыть тестами эндпоинты API на корректность фильтрации и пагинации.

### 📂 Рекомендуемая структура проекта

```text
spacenews_project/
│
├── core/                    # Настройки проекта
│   ├── __init__.py          # Инициализация Celery
│   ├── celery.py            # Конфигурация Celery
│   ├── settings.py
│   └── urls.py
│
├── news/                    # Приложение новостей
│   ├── migrations/
│   ├── api/
│   │   ├── serializers.py   # Сериализаторы DRF
│   │   ├── views.py         # ViewSets для эндпоинтов
│   │   └── urls.py          # Роуты API (/api/v1/news/)
│   ├── models.py            # Модели (Article, Category)

```

### 🚀 Дополнительные задания "со звездочкой" (Level Up)

1. **Docker Compose:** Написать конфигурацию для локального запуска всего стека (Django, Celery Worker, Celery Beat, Redis, PostgreSQL) одной командой `docker-compose up`.
2. **Логирование и мониторинг:** Настроить отправку логов о сбоях импорта в консоль и файлы, а также интегрировать простой инструмент мониторинга задач Celery (например, Flower).
